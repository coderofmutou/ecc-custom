# upstream-sync 设计说明

## 目录结构

```text
.claude/skills/upstream-sync/
├── SKILL.md                 # 用户层面的 skill 介绍(何时用、怎么用)
├── README.md                # 本文件:设计说明、安全约束的理由
└── scripts/
    ├── check-upstream.js     # 只读检查,不修改任何东西
    └── sync-upstream.js      # 唯一会修改工作区的脚本,只在新分支上操作
```

## 为什么单独建一个技能,不塞进 dependency-graph

两者关注点不同:dependency-graph 管"仓库内部组件之间的引用关系",upstream-sync 管"和上游仓库的同步流程"。高内聚低耦合——但两者会互相配合:upstream-sync 只读引用 dependency-graph 已导出的函数(`buildGraph`/`loadLedger`/`resolveComponentPath`/`buildPrunePlan`/`applyPrunePlan`),从不修改 dependency-graph 目录下的任何文件。

## 为什么是 merge 而不是 rebase

这个 fork 的 `main` 分支会被推到远端(`origin`),属于"公共历史"。`rebase` 会改写提交历史,需要 force-push 才能同步到远端,而用户的全局规则里 force push 前必须显式确认——用 `merge` 完全不需要改写历史,不会碰到这个问题。

## 为什么只在新分支上操作,不直接改 main

同步上游是一个"探索性"操作——合并前无法 100% 确定会不会有冲突、冲突大不大。在 `sync/upstream-<date>` 分支上做,即使合并出了问题,`main` 分支毫发无损,随时可以 `git branch -D sync/upstream-<date>` 重来。合并到 `main` 这一步永远交还给用户自己执行,本技能不做,也绝不 `push`——这是删除/合并类操作需要人工确认的红线。

## 冲突分类:为什么"命中已裁剪路径"可以机械化处理

如果一个文件路径对应 `decisions.json` 里某条 `decision=exclude` 的记录,说明**不管上游把这个文件改成什么样,你都不打算要它**——所以处理这类冲突不需要看 diff 内容,直接 `git rm -f` 就是正确答案。这类冲突不会随着仓库演进累积:每次上游改动都可能重新触发,但每次都是同样确定性的处理方式,脚本化之后成本几乎为零。

真正需要人工/Claude 判断的,只剩下"你决定保留、而且上游和你本地都改过同一处"的文件——这个集合天然是有限且可控的,不会随时间无限增长。

## 为什么合并完成后要无条件重跑一次裁剪,而且立刻提交

git 的合并冲突检测只在"同一个位置被双方修改"时触发。如果上游在一个已经被你整体删除的目录下**新增**了一个全新文件,`git merge` 不会报冲突(对 git 来说这是一次干净的"添加"),但这个新文件仍然落在你已经决定排除的范围内。合并完成后无条件重跑一次 `prune`,就是为了兜底这种"新增不触发冲突,但确实不该留下"的情况。

这批补删和"命中已裁剪路径的合并冲突"性质完全一样——都是机械化、确定性的操作,不需要人工看内容——所以脚本会在补删完成后立刻 `git add -A && git commit`,不留任何未提交的删除。早期版本这里漏了提交这一步,补删的文件会以未提交状态滞留在 sync 分支上,真去执行 SKILL.md 最后一步 `git checkout main` 时可能被 git 挡下,或者把这批改动当成游离的工作区状态悄悄带过去——这违背了"新分支上操作、随时能重来"的安全前提,现在已经修复。

注意:这个自动提交只覆盖脚本自己做的补删。SKILL.md 第 4 步(dependency-graph 的 registry 刷新)是你/Claude 在脚本运行结束后手动跑的,`sync-upstream.js` 对此不知情,所以不会替你提交——SKILL.md 第 5 步专门补了这个提交动作,别漏掉。

## 为什么合并被人工冲突挡住时,不自动重跑裁剪

如果还有 `manual` 冲突没解决,说明这次 `git merge` 还处于"进行中"状态(`MERGE_HEAD` 还存在,索引里还有未合并的条目)。这时候如果再往工作区里叠加一批裁剪产生的文件删除,会让本来就复杂的冲突处理现场更难看清楚哪些改动来自合并本身、哪些来自裁剪。所以脚本选择在这种情况下什么都不做,把"合并完成后要不要重新裁剪"留给用户自己在处理完人工冲突、提交合并之后手动决定。

## 已知局限

- 不处理"从 `main` 派生同步分支"之外的场景——如果当前不在 `main` 分支,脚本会直接报错退出,不会猜测你想干什么。
- 分支名 `sync/upstream-<date>` 如果已经存在(比如同一天重跑),`git checkout -b` 会报错。这是有意为之——脚本不会自动 `-D` 删除旧分支重建,避免误删你可能还想保留的上一次同步现场。需要重跑的话,自己先确认旧分支可以丢弃,再手动删除。
- 冲突分类只按"文件路径是否命中 `decisions.json` 里的已裁剪路径"这一个信号,不理解 diff 内容本身。如果一个文件路径没有对应任何决策记录,哪怕它的冲突内容其实很简单,也会被归入需要人工处理的一类——这是保守的选择,宁可多留给人工看一眼,也不猜测。

## 自测

本技能没有独立的自测脚本——`check-upstream.js`/`sync-upstream.js` 都依赖真实的 git 仓库状态和 `upstream` 远程,不适合用 dependency-graph 那种临时 fixture 目录的方式做单元测试。验证方式是实际跑一次 `check-upstream.js`(只读,随时可以跑);`sync-upstream.js` 建议先在确实有上游变动、或者手动构造的小范围测试场景下跑一次,确认冲突分类符合预期。
